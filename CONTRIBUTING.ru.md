## Кодекс поведения

Этот проект и все участвующие в нём руководствуются уважением и профессионализмом. Будьте добры, будьте конструктивны и помогите нам создать потрясающий инструмент вместе.

## Начало работы

1. Сделайте форк репозитория
2. Клонируйте свой форк: `git clone https://github.com/YOUR-USERNAME/lazyvimx.git`
3. Создайте новую ветку: `git checkout -b feature/your-feature-name`
4. Внесите изменения
5. Тщательно протестируйте изменения
6. Зафиксируйте изменения (см. [Сообщения коммитов](#сообщения-коммитов))
7. Отправьте в свой форк: `git push origin feature/your-feature-name`
8. Откройте Pull Request

## Как я могу внести свой вклад?

### Сообщения об ошибках

Перед созданием отчёта об ошибке проверьте существующие issues, чтобы избежать дубликатов. При создании отчёта об ошибке включите:

- **Чёткий заголовок**, описывающий проблему
- **Детальное описание** проблемы
- **Шаги для воспроизведения** поведения
- **Ожидаемое поведение** vs фактическое поведение
- **Скриншоты**, если применимо
- **Детали окружения**: версия Neovim, ОС, версия lazyvimx
- **Конфигурация**: соответствующие части вашей конфигурации

### Предложения по улучшению

Предложения по улучшению отслеживаются как GitHub issues. При создании предложения по улучшению включите:

- **Чёткий заголовок**, описывающий улучшение
- **Детальное описание** предлагаемой функциональности
- **Варианты использования** и примеры того, как это будет использоваться
- **Альтернативные подходы**, которые вы рассматривали
- **Дополнительный контекст** или скриншоты

### Добавление extras

Extras — это опциональные модули функций, которые расширяют LazyVim. Они должны:

- Быть **опциональными** и не влиять на базовую функциональность
- Следовать **существующим паттернам** из других extras
- Включать **документацию** в EXTRAS.md
- Быть **правильно категоризированными** (ui, coding, motions и т.д.)
- Быть **протестированными**, чтобы убедиться, что они работают корректно

### Добавление overrides

Overrides изменяют существующие конфигурации плагинов. Они должны:

- Использовать `optional = true` для корректной деградации
- Не нарушать функциональность, если плагин не установлен
- Быть хорошо задокументированы с комментариями
- Следовать существующим паттернам

### Улучшение документации

Улучшения документации всегда приветствуются:

- Исправление опечаток или уточнение непонятных разделов
- Добавление примеров или вариантов использования
- Перевод контента (особенно на русский)
- Обновление устаревшей информации
- Добавление отсутствующей документации

## Настройка окружения для разработки

### Требования

- Neovim >= 0.10.0
- Git
- Базовое понимание Lua и LazyVim

### Локальная разработка

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/aimuzov/lazyvimx.git ~/.local/share/nvim/lazy/lazyvimx
   ```

2. Создайте тестовую конфигурацию:
   ```lua
   -- ~/.config/nvim-test/init.lua
   local lazy_path = vim.fn.stdpath("data") .. "/lazy/lazy.nvim"
   vim.opt.rtp:prepend(lazy_path)

   require("lazy").setup({
     spec = {
       { dir = "~/.local/share/nvim/lazy/lazyvimx", import = "lazyvimx.boot" },
       { import = "lazyvimx.extras.core.all" },
     },
   })
   ```

3. Протестируйте изменения:
   ```bash
   NVIM_APPNAME=nvim-test nvim
   ```

## Создание extras

### Шаблон модуля extra

```lua
-- lua/lazyvimx/extras/category/my-extra.lua
return {
  {
    "plugin/name",
    opts = {
      -- Опции конфигурации
    },
    keys = {
      -- Привязки клавиш
    },
    dependencies = {
      -- Зависимости
    },
  },
}
```

### Шаги для создания extra

1. **Выберите категорию**: ui, coding, motions, buf, git, lang, linting, ai, dap, perf или test
2. **Создайте файл**: `lua/lazyvimx/extras/category/extra-name.lua`
3. **Напишите модуль**: Следуйте шаблону выше
4. **Добавьте в реестр**: Включите в `lua/lazyvimx/extras/core/extras.lua`
5. **Задокументируйте**: Добавьте запись в `docs/EXTRAS.md` с описанием, функциями и использованием
6. **Протестируйте**: Убедитесь, что extra работает корректно
7. **Обновите счётчик**: Увеличьте количество extras в README, если добавляете новый extra

### Руководство по extras

- **Именование**: Используйте kebab-case для имён файлов (например, `better-diagnostic.lua`)
- **Опциональность**: Всегда делайте плагины опциональными с `optional = true`
- **Зависимости**: Явно объявляйте все зависимости
- **Условия**: Используйте `cond` для условной загрузки
- **Документация**: Включайте встроенные комментарии для сложной логики

## Создание overrides

### Шаблон модуля override

```lua
-- lua/lazyvimx/overrides/category/my-override.lua
return {
  {
    "plugin/name",
    optional = true,
    opts = function(_, opts)
      -- Изменение существующих opts
      opts.new_option = value
      return opts
    end,
  },
}
```

### Руководство по overrides

- **Опциональность**: Всегда используйте `optional = true`
- **Безопасность**: Проверяйте существование плагина перед изменением
- **Документированность**: Добавляйте комментарии, объясняющие переопределение
- **Категоризация**: Размещайте в правильной категории (lazyvim, snacks, bufferline, other)

## Руководство по написанию кода

### Стиль Lua

- **Отступы**: Табы (ширина 4)
- **Кавычки**: Двойные кавычки для строк
- **Длина строки**: Максимум 120 символов
- **Комментарии**: Объясняйте "почему", а не "что"

### Хорошие практики

- **DRY**: Не повторяйтесь
- **KISS**: Держите всё просто
- **Тестирование**: Тестируйте все изменения
- **Документация**: Документируйте все публичные API
- **Обратная совместимость**: Не ломайте существующие конфигурации

### Пример

```lua
-- Хорошо
local function get_theme()
  local util = require("lazyvimx.util.general")
  return util.theme_is_dark() and "dark" or "light"
end

-- Плохо
local function get_theme()
  if require("lazyvimx.util.general").theme_is_dark() == true then
    return "dark"
  else
    return "light"
  end
end
```

## Сообщения коммитов

Следуйте [Conventional Commits](https://www.conventionalcommits.org/):

### Формат

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Типы

- `feat`: Новая функция
- `fix`: Исправление ошибки
- `docs`: Изменения в документации
- `style`: Изменения стиля кода (форматирование и т.д.)
- `refactor`: Рефакторинг кода
- `perf`: Улучшения производительности
- `test`: Добавление или обновление тестов
- `chore`: Задачи обслуживания

### Примеры

```bash
feat(ui): add better-diagnostic extra
fix(motions): resolve langmapper conflict with which-key
docs(readme): update installation instructions
refactor(util): simplify color blending function
```

## Процесс создания Pull Request

1. **Обновите документацию** для любых изменений, видимых пользователю
2. **Обновите CHANGELOG.md** в разделе "Unreleased"
3. **Обновите счётчик extras**, если вы добавили/удалили extras
4. **Тщательно протестируйте** в чистом экземпляре Neovim
5. **Убедитесь, что коммиты** следуют формату conventional commits
6. **Свяжите issues**, которые решает PR
7. **Запросите ревью** у мейнтейнеров
8. **Обрабатывайте обратную связь** оперативно и профессионально

### Шаблон PR

```markdown
## Описание
Краткое описание изменений

## Тип изменения
- [ ] Исправление ошибки
- [ ] Новая функция
- [ ] Обновление документации
- [ ] Рефакторинг

## Чек-лист
- [ ] Документация обновлена
- [ ] CHANGELOG.md обновлён
- [ ] Протестировано в чистом окружении
- [ ] Следует руководству по написанию кода
- [ ] Коммиты следуют conventional commits

## Связанные Issues
Closes #<issue_number>
```
